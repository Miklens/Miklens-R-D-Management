# Database Design (Firestore Schema)

## Overview
The database uses Google Cloud Firestore, a NoSQL document database. The schema is designed for high read scalability, fast querying for dashboards, and offline synchronization capabilities.

All documents across all collections implicitly include the following base metadata fields:
- `createdAt` (Timestamp)
- `updatedAt` (Timestamp)
- `createdBy` (Reference/UID)
- `updatedBy` (Reference/UID)
- `isDeleted` (Boolean, for soft delete support)
- `version` (Number, for optimistic concurrency and tracking)

## Collections

### 1. `users`
Stores user profile information and RBAC (Role-Based Access Control) details. Authentication is handled by Firebase Auth, but user metadata lives here.
- `id` (String - Matches Firebase Auth UID)
- `email` (String)
- `firstName` (String)
- `lastName` (String)
- `roleId` (Reference to `roles`)
- `departmentId` (Reference to `departments`)
- `designation` (String)
- `skills` (Array of Strings)
- `experienceYears` (Number)
- `isActive` (Boolean)

### 2. `roles`
Defines system roles and their permissions.
- `id` (String)
- `name` (String - Admin, Management, Scientist)
- `permissions` (Array of Strings - e.g., 'read:reports', 'write:logs')

### 3. `departments`
- `id` (String)
- `name` (String)
- `headId` (Reference to `users`)

### 4. `products`
The core entity representing a research product.
- `id` (String)
- `name` (String)
- `category` (String)
- `description` (String)
- `targetMarket` (String)
- `currentStage` (String - e.g., 'Lab Testing', 'Field Trial')
- `assignedTeam` (Array of References to `users`)
- `timeline` (Object - {startDate, targetDate, completionDate})
- `budget` (Object - {allocated, spent})
- `risks` (Array of Objects - {description, severity, status})
- `dependencies` (Array of Strings)
- `aiSummary` (String - Generated by AI)
- `status` (String - 'Active', 'Delayed', 'Blocked', 'Completed')

### 5. `projects`
Projects can group multiple products or represent cross-functional initiatives.
- `id` (String)
- `name` (String)
- `description` (String)
- `managerId` (Reference to `users`)
- `status` (String)

### 6. `experiments`
Every experiment belongs to a product.
- `id` (String)
- `productId` (Reference to `products`)
- `experimentNumber` (String)
- `version` (Number)
- `objective` (String)
- `hypothesis` (String)
- `materials` (Array of Strings)
- `procedure` (String)
- `results` (Object - {outcome: 'Success' | 'Failure' | 'Partial', rootCause, nextAction})
- `aiAnalysis` (String)
- `leadScientistId` (Reference to `users`)

### 7. `daily_logs`
The primary daily entry by scientists.
- `id` (String)
- `userId` (Reference to `users`)
- `date` (Timestamp)
- `productId` (Reference to `products`)
- `experimentId` (Reference to `experiments`)
- `todaysObjective` (String)
- `activitiesPerformed` (String)
- `observations` (String)
- `problems` (String)
- `achievements` (String)
- `nextSteps` (String)
- `timeSpentMinutes` (Number)
- `completionStatus` (String)
- `estimatedProductStage` (String)
- `confidenceLevel` (Number 1-100)
- `aiNotes` (String)

### 8. Reports (`weekly_reports`, `monthly_reports`, `yearly_reports`)
Generated mostly by AI from aggregating daily logs.
- `id` (String)
- `type` (String - 'Employee', 'Product', 'Department', 'Executive')
- `targetId` (Reference to respective entity)
- `periodStart` (Timestamp)
- `periodEnd` (Timestamp)
- `summary` (String)
- `metrics` (Object - various KPIs)
- `aiInsights` (String)

### 9. `tasks` / `milestones`
- `id` (String)
- `title` (String)
- `description` (String)
- `assignedTo` (Reference to `users`)
- `productId` (Reference to `products`)
- `dueDate` (Timestamp)
- `status` (String)
- `isMilestone` (Boolean)

### 10. File Collections (`documents`, `images`, `videos`)
Metadata for files stored in Firebase Storage.
- `id` (String)
- `relatedEntityType` (String - 'product', 'experiment', 'daily_log')
- `relatedEntityId` (String)
- `fileName` (String)
- `fileUrl` (String)
- `fileType` (String)
- `sizeBytes` (Number)

### 11. Support Collections
- `comments`: Threaded discussions on logs, experiments, or products.
- `notifications`: Alerts for users (mentions, approvals, AI warnings).
- `calendar`: Events, deadlines, field trial schedules.
- `audit_logs`: Immutable tracking of critical changes.
- `settings`: System-wide or user-specific configurations.
- `activity_logs`: Feed of what happened recently for the dashboard.

## Database Security Rules Strategy
- **Authentication:** Only authenticated users can read/write.
- **RBAC Enforcement:** Rules will check custom claims or the user's document in the `users` collection. Admin has global read/write. Management has global read, limited write. Scientists can read their assigned products/experiments and read/write their own daily logs.
- **Data Validation:** Basic type checking and required fields enforced within Firestore Security Rules to prevent malformed data.
